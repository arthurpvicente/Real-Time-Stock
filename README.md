## 1. Overview

Signalist is a production-style stock tracking web application built with Next.js 15 and the App Router. It gives investors a single place to monitor the market, manage a personal watchlist, track holdings with real-time gain/loss calculations, and receive AI-curated news summaries directly to their inbox.

Key value propositions:
- **Real-time market data** via Finnhub API with TradingView embedded charts
- **Portfolio analytics** — track shares, buy price, and live gain/loss across all holdings
- **AI-powered emails** — personalized welcome messages and daily news digests generated by Google Gemini
- **Secure, session-based auth** — email/password login with investor profile customization

## 3. Features

### Authentication
- Email/password sign-up and sign-in via Better Auth
- Investor profile collected at registration: country, investment goals, risk tolerance, preferred industry
- Profile picture upload via URL
- Session stored in HTTP-only cookies with middleware-enforced route protection

### Dashboard
- **Popular Stocks Grid** — real-time prices and % change for a curated list of major stocks
- **Market Overview Widget** — TradingView market snapshot with Financial, Technology, and Services tabs
- **Sector Heatmap** — interactive visualization of sector performance with size and color encoding
- **Market News Timeline** — top market stories via TradingView's news widget

### Stock Search
- **Cmd+K (Mac) / Ctrl+K (Windows) command palette** — opens a global search dialog from anywhere in the app
- Debounced input for efficient API calls
- Results show exchange and stock type alongside symbol
- Star toggle reflects live portfolio state — starred stocks are shown as already tracked

### Stock Detail Page
- Dynamic route `/stocks/[symbol]` for any tracked or searched stock
- Embedded TradingView widgets: candlestick chart, baseline chart, technical analysis, company profile, and company financials
- Add to Portfolio / Add Holding buttons directly on the stock page

### Portfolio
- Add and remove stocks from a personalized portfolio (`/portfolio` page)
- Portfolio page shows all tracked stocks with live price and % change
- Star button in search results reflects real-time portfolio membership
- **Holdings tracking** — record shares purchased, buy price, and buy date per stock
- **Real-time gain/loss** — calculated against live Finnhub quote on every load
- **Portfolio summary** — total amount invested, current portfolio value, and overall return percentage
- **Auto-refresh every 20 seconds** — keeps gain/loss figures current without a manual reload
- Add/Edit Holdings dialog with automatic share calculation from investment amount

### AI-Powered Emails
- **Welcome email** — triggered on sign-up, content personalized to the user's investor profile using Google Gemini 1.5 Flash
- **Daily news summary** — sent every day at noon, curated per user based on their watchlist stocks and summarized by Gemini AI

### Background Jobs (Inngest)
- Event-driven job queue decouples email sending from the request cycle
- Sign-up does not block on email delivery — failures are logged and swallowed gracefully
- Cron-scheduled daily digest runs independently of any user session

## 7. Getting Started

### Prerequisites

- Node.js 18+
- [MongoDB Atlas](https://www.mongodb.com/atlas): account (free tier)
- [Finnhub](https://finnhub.io/): API key (free tier)
- [Resend](https://resend.com/): account and API key
- [Google AI Studio](https://aistudio.google.com/): Gemini API key
- [Inngest](https://inngest.com/): background job processing service — it lets you run tasks outside of 
  the normal request cycle, so the user doesn't have to wait for them to finish.

### Installation

```bash
git clone https://github.com/arthurpvicente/Real-Time-Stock.git
cd Real-Time-Stock
npm install
```

### Environment Variables

Create a `.env` file in the root of the project:

```env
NODE_ENV='development'
NEXT_PUBLIC_BASE_URL=http://localhost:3000

# Finnhub (get at https://finnhub.io/dashboard)
NEXT_PUBLIC_FINNHUB_API_KEY=your_public_key
FINNHUB_BASE_URL=https://finnhub.io/api/v1

# MongoDB Atlas connection string
MONGODB_URI=mongodb+srv://<user>:<password>@cluster.mongodb.net/<dbname>

# Better Auth (generate a random 32+ character string for the secret)
BETTER_AUTH_SECRET=your_random_secret
BETTER_AUTH_URL=http://localhost:3000

# Google Gemini AI (https://aistudio.google.com/apikey)
GEMINI_API_KEY=your_gemini_key

# Resend email service (https://resend.com/api-keys)
RESEND_API_KEY=your_resend_key

# Inngest background jobs (https://app.inngest.com/settings/signing-keys)
INNGEST_SIGNING_KEY=your_inngest_signing_key
```

> `NEXT_PUBLIC_FINNHUB_API_KEY` is exposed to the browser. Set usage limits on the Finnhub dashboard to prevent abuse.

### Running the App

Start the Next.js development server:

```bash
npm run dev
```

In a **separate terminal**, start the Inngest local dev server to handle background jobs (welcome emails, daily news):

```bash
npx inngest-cli@latest dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

## 8. Database Schema

### Portfolio

Stored in the `portfolios` collection. Each document represents one stock in a user's portfolio.

| Field | Type | Constraints |
|-------|------|-------------|
| `userId` | String | required, indexed |
| `symbol` | String | required, uppercase, indexed |
| `company` | String | required |
| `addedAt` | Date | defaults to `Date.now` |

Unique compound index: `{ userId, symbol }` — prevents duplicate entries per user.

### Holdings

Stored in the `holdings` collection. Each document represents a position a user holds in a stock.

| Field | Type | Constraints |
|-------|------|-------------|
| `userId` | String | required, indexed |
| `symbol` | String | required, uppercase, indexed |
| `company` | String | required |
| `shares` | Number | required, min 0.0001 |
| `buyPrice` | Number | required, min 0.0001 |
| `buyDate` | Date | required |

Unique compound index: `{ userId, symbol }` — one holding record per stock per user.

### Users

Managed automatically by Better Auth. Stored in MongoDB with fields: `id`, `name`, `email`, `image`, `country`, `createdAt`, `updatedAt`. Sessions are stored as a separate collection.

## 9. Server Actions Reference

All data mutations and fetches go through Next.js Server Actions (`'use server'`). There are no public REST API routes other than the Inngest webhook.

**Authentication** (`lib/actions/auth.actions.ts`)

| Action | Description |
|--------|-------------|
| `signUpWithEmail` | Register new user, fire `app/user.created` Inngest event |
| `signInWithEmail` | Authenticate and establish session |
| `signOut` | Destroy session cookie |
| `updateUserImage` | Update profile picture URL |

**Stock Data** (`lib/actions/finnhub.actions.ts`)

| Action | Description |
|--------|-------------|
| `searchStocks` | Search symbols via Finnhub `/search` |
| `searchStocksForUser` | Search + annotate each result with watchlist membership |
| `getStockQuote` | Real-time price and % change (no-store cache) |
| `getStockProfile` | Company name, logo, exchange (1-hour cache) |
| `getNews` | Market or company-specific news (5-minute cache) |

**Portfolio** (`lib/actions/portfolio.actions.ts`)

| Action | Description |
|--------|-------------|
| `addToPortfolioAction` | Add stock to user's watchlist |
| `removeFromPortfolioAction` | Remove stock from watchlist |
| `getPortfolioItems` | Fetch all watchlist stocks for current user |
| `getPortfolioSymbolsByEmail` | Get symbols list only (used in news job) |

**Holdings** (`lib/actions/holdings.actions.ts`)

| Action | Description |
|--------|-------------|
| `addHoldingAction` | Record a stock position (shares, buy price, date) |
| `removeHoldingAction` | Delete a holding record |
| `getHoldings` | Fetch all holdings, enriched with live quote data |

## 10. Background Jobs

Managed by Inngest. Jobs are defined in `lib/inngest/functions.ts` and triggered via events or cron schedule.

| Job | Trigger | What It Does |
|-----|---------|-------------|
| `sendSignUpEmail` | `app/user.created` event (fires on registration) | Calls Gemini to generate a personalized welcome message based on the user's investor profile, then sends it via Resend |
| `sendDailyNewsSummary` | Cron `0 12 * * *` (daily at noon UTC) | For each user: fetches portfolio, retrieves relevant news from Finnhub, summarizes via Gemini, sends personalized email via Resend |

The Inngest webhook endpoint is at `/api/inngest` and handles `GET`, `POST`, and `PUT` methods.

## 11. Troubleshooting

### Error: `MONGODB_URI must be set within .env` (Vercel build failure)

```
MONGODB_URI must be set within .env
> Build error occurred
[Error: Failed to collect page data for /] { type: 'Error' }
```

**Why it happens:** The auth instance initializes at module load time during the build. If `MONGODB_URI` is missing from Vercel's environment, the build crashes — not just at runtime.

**Fix:**
1. Go to Vercel dashboard → Project → Settings → Environment Variables
2. Add all required variables: `MONGODB_URI`, `NEXT_PUBLIC_FINNHUB_API_KEY`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `RESEND_API_KEY`, `GEMINI_API_KEY`, `INNGEST_SIGNING_KEY`
3. Redeploy the project

---

### Error: Sign-up blocked when Inngest is unavailable

**What happened:** `inngest.send()` was called without error handling inside the sign-up action. If Inngest was down or misconfigured, it threw an unhandled exception and blocked account creation entirely.

**Fix:** Wrapped the `inngest.send()` call in a try/catch. Welcome email failure is now logged but does not interrupt sign-up.

---

### Error: Wrong watchlist state shown in search results (`isInWatchlist`)

**What happened:** The star button in search results showed incorrect starred/unstarred state — stocks already in the watchlist appeared unstarred and vice versa.

**Fix:** Corrected the `isInWatchlist` logic in the search results server action to properly compare stock symbols against the user's watchlist.

---

### Error: Auth errors not returned to the sign-up form

**What happened:** `auth.actions.ts` was catching errors silently and not returning them, so users saw no feedback when sign-up failed (e.g., email already in use).

**Fix:** Updated the action to return real error messages from Better Auth so the form can display them.

---

### Error: Unauthenticated users could access protected routes

**What happened:** Middleware was not redirecting users without a session to `/sign-in`, so protected pages were accessible without logging in.

**Fix:** Updated `middleware/index.ts` to check session state and redirect unauthenticated users.

---

### Error: Sign-up form validation bugs

**What happened:** Multiple issues in the sign-up form:
- Label showed "Email name is required" instead of the correct field name
- Broken regex for field validation
- Invalid `message` prop passed to a component that didn't accept it
- Unused `signUpWithEmail` import causing lint/build warnings

**Fix:** Each issue was resolved individually — corrected labels, fixed regex, removed invalid props, and cleaned up unused imports.

---

### Error: Inngest signing key mismatch in production

**What happened:** The `/api/inngest` endpoint rejects requests that do not match the expected signing key, causing background jobs to silently fail in production.

**Fix:** Ensure `INNGEST_SIGNING_KEY` in your Vercel environment variables exactly matches the signing key shown in your Inngest dashboard under **Settings → Signing Keys**. After updating, redeploy and re-sync the Inngest app endpoint.
